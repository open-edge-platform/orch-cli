### Implementation Pattern:

Each standardized command follows this pattern:

1. **Template Constants:**
   ```go
   const (
       DEFAULT_RESOURCE_FORMAT = "table{{.Field1}}\t{{.Field2}}"
       DEFAULT_RESOURCE_VERBOSE_FORMAT = "Field1: {{.Field1}}\nField2: {{.Field2}}"
       RESOURCE_OUTPUT_TEMPLATE_ENVVAR = "ORCH_CLI_RESOURCE_OUTPUT_TEMPLATE"
   )
   ```

2. **Print Helper Function:**
   ```go
   func printResources(cmd *cobra.Command, writer io.Writer, resources []Resource, 
                       orderBy *string, outputFilter *string, verbose bool) error {
       outputFormat := getResourceOutputFormat(cmd, verbose)
       outputType, _ := cmd.Flags().GetString("output-type")
       
       result := CommandResult{
           Format:    format.Format(outputFormat),
           Filter:    *outputFilter,
           OrderBy:   *orderBy,
           OutputAs:  toOutputType(outputType),
           Data:      resources,
       }
       
       GenerateOutput(writer, &result)
       return nil
   }
   ```

3. **Order-By Validation:**
   ```go
   func getValidatedResourceOrderBy(ctx, cmd, client, project) (*string, error) {
       raw, _ := cmd.Flags().GetString("order-by")
       outputType, _ := cmd.Flags().GetString("output-type")
       
       if outputType == "table" {
           return normalizeOrderByForClientSorting(raw, ResourceType{})
       }
       return normalizeOrderByWithAPIProbe(raw, "resources", ResourceType{}, apiProbe)
   }
   ```

4. **Standard Flags:**
   ```go
   cmd.Flags().StringP("filter", "f", "", "API filter")
   cmd.Flags().String("order-by", "", "order results by field")
   addStandardListOutputFlags(cmd)  // Adds output-type, output-filter, output-template, etc.
   ```

## Part 2: Removed Unnecessary Flattening
Previous analysis identified that some resources had unnecessary flattening:
- **OS Profiles**: Flat API struct - flattening UNNECESSARY
- **Groups**: 3 string fields - flattening UNNECESSARY (was 1:1 copy!)
- **Users**: 5 strings + 1 bool pointer - flattening UNNECESSARY

Resources that justified flattening:
- **Projects**: 3-level nested structs (`{{.Status.ProjectStatus.StatusIndicator}}`)
- **Organizations**: 3-level nested structs (`{{.Status.OrgStatus.StatusIndicator}}`)

## Key Decisions

1. **When to flatten**: Only for deeply nested structs (3+ levels deep)
   - Projects/Organizations: Keep flattening (justified)
   - OS Profiles/Groups/Users: Remove flattening (unnecessary)

2. **Template approach**: Use `str` helper function for pointer fields
   - Before: `{{.Name}}` (required flattening to convert `*string` to `string`)
   - After: `{{str .Name}}` (handles `*string` safely in template)

3. **User fields**: Email, FirstName, LastName are `string` not `*string`
   - Fixed by removing `none` helper: `{{.Email}}` instead of `{{none .Email}}`

4. **UserInspectItem**: Kept for get command only
   - Reason: Aggregates data from multiple sources (user info + groups + roles)
   - Not used for list command (which now uses API struct directly)
